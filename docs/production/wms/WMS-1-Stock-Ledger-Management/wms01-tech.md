# Technical Specifications: Stock Ledger Management (WMS-1)

## 1. System Architecture
- **Frontend**: Next.js 15 Server Components for fast data fetching and rendering of the ledger table.
- **Backend**: Next.js API Routes (Server Actions for mutations).
- **Database**: PostgreSQL (Neon) using Drizzle ORM.
- **Event Bus**: MediatR pattern for internal domain events (e.g., broadcasting stock movements to Finance).

## 2. Database Schema

### Table: `inventory_items`
| Column Name | Type | Constraints | Description |
|-------------|------|-------------|-------------|
| `id` | UUID | PRIMARY KEY | Unique identifier |
| `sku` | VARCHAR(100) | UNIQUE, NOT NULL | Stock Keeping Unit |
| `name` | VARCHAR(255) | NOT NULL | Item name |
| `uom` | VARCHAR(50) | NOT NULL | Unit of Measure (e.g., pcs, kg) |
| `tenant_id` | UUID | NOT NULL, FK | Multi-tenancy isolation |

### Table: `locations`
| Column Name | Type | Constraints | Description |
|-------------|------|-------------|-------------|
| `id` | UUID | PRIMARY KEY | Unique identifier |
| `code` | VARCHAR(100) | UNIQUE, NOT NULL | Location code (e.g., WH-A-01) |
| `type` | VARCHAR(50) | NOT NULL | Enum: WAREHOUSE, TRANSIT, SUPPLIER, CUSTOMER |
| `tenant_id` | UUID | NOT NULL, FK | Multi-tenancy isolation |

### Table: `stock_ledger`
| Column Name | Type | Constraints | Description |
|-------------|------|-------------|-------------|
| `id` | UUID | PRIMARY KEY | Unique identifier |
| `transaction_id`| UUID | NOT NULL | Grouping ID for double-entry pairs |
| `item_id` | UUID | NOT NULL, FK | Reference to `inventory_items` |
| `location_id` | UUID | NOT NULL, FK | Reference to `locations` |
| `quantity` | DECIMAL | NOT NULL | Positive (Debit) or Negative (Credit) |
| `unit_cost` | DECIMAL | NOT NULL | Cost per unit at time of transaction |
| `reference_type`| VARCHAR(50) | NOT NULL | e.g., PURCHASE_RECEIPT, SHIPMENT |
| `reference_id` | UUID | NOT NULL | Link to the source document |
| `tenant_id` | UUID | NOT NULL, FK | Multi-tenancy isolation |
| `created_at` | TIMESTAMP | DEFAULT NOW() | Immutable timestamp |

## 3. Drizzle ORM Schema Snippet
```typescript
import { pgTable, uuid, varchar, decimal, timestamp, index } from "drizzle-orm/pg-core";

export const inventoryItems = pgTable("inventory_items", {
  id: uuid("id").primaryKey().defaultRandom(),
  sku: varchar("sku", { length: 100 }).notNull().unique(),
  name: varchar("name", { length: 255 }).notNull(),
  uom: varchar("uom", { length: 50 }).notNull(),
  tenantId: uuid("tenant_id").notNull(),
});

export const locations = pgTable("locations", {
  id: uuid("id").primaryKey().defaultRandom(),
  code: varchar("code", { length: 100 }).notNull().unique(),
  type: varchar("type", { length: 50 }).notNull(),
  tenantId: uuid("tenant_id").notNull(),
});

export const stockLedger = pgTable("stock_ledger", {
  id: uuid("id").primaryKey().defaultRandom(),
  transactionId: uuid("transaction_id").notNull(), // Links Debit and Credit lines
  itemId: uuid("item_id").references(() => inventoryItems.id).notNull(),
  locationId: uuid("location_id").references(() => locations.id).notNull(),
  quantity: decimal("quantity").notNull(),
  unitCost: decimal("unit_cost").notNull(),
  referenceType: varchar("reference_type", { length: 50 }).notNull(),
  referenceId: uuid("reference_id").notNull(),
  tenantId: uuid("tenant_id").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
}, (table) => {
  return {
    itemLocIdx: index("item_loc_idx").on(table.itemId, table.locationId),
    tenantIdx: index("tenant_idx").on(table.tenantId)
  }
});
```

## 4. API Endpoints

### GET `/api/v1/wms/stock-ledger`
- **Query Params**: `sku`, `locationId`, `startDate`, `endDate`
- **Response**: Array of ledger entries with calculated running balance.

### GET `/api/v1/wms/inventory/balance`
- **Query Params**: `itemId`, `locationId`
- **Description**: Calculates sum of quantity from ledger for the specific item/location.

## 5. Domain Events
- **Raised**:
  - `StockMovement` (ItemID, LocationID, Qty, Cost, RefType) -> Consumed by Finance to create General Ledger entries.
- **Consumed**:
  - None for the ledger directly (it is the source of truth).

## 6. Permissions (RBAC)
- `wms.ledger.view`: Allowed to view the ledger and balances.
- *Note: Direct write access to the ledger via API is restricted. Entries are strictly generated by operational events (Putaway, Pick, Adjust).*

## 7. Performance Considerations
- The balance calculation `SUM(quantity)` over millions of rows could be slow. We will use a Materialized View or an indexed `inventory_balances` rollup table that is updated transactionally alongside the ledger insert.

## 8. Security Considerations
- **Immutable Ledger**: The database role used by the application should NOT have `UPDATE` or `DELETE` privileges on the `stock_ledger` table. Corrections must be `INSERT`s of opposite values.
- RLS (Row-Level Security) must be enabled on `tenant_id`.

## 9. Error Handling
- Use database transactions (`db.transaction()`) to ensure both Debit and Credit lines are written together. If one fails, rollback both.
